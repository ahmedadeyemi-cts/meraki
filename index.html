Lifecycle Thresholds

  • Use: 100+ weekly calls and at least 75% success
  • Engage: 5,000+ weekly calls and at least 85% success
  • Adopt: 7,500+ weekly calls and at least 95% success

Recommended Cron Cadence

  • Every 6 hours for steady call growth toward 8,000 calls in 2 weeks
  • Use manual Run Now for testing or troubleshooting
  • Use force live refresh only when you need current data immediately

SLA Thresholds

  • Green: success rate at or above 95%
  • Yellow: success rate between 85% and 94%
  • Red: success rate below 85%

Adoption Runbook Guidance

              

                
Use a six-hour cron cadence for steady API generation without creating unnecessary noise.

                
Run a force-live execution only when you need current Meraki state immediately for validation or executive proof.

                
Keep write-test mode disabled unless you have an approved test network ID for the customer.

                
Use the proof report to show both raw call volume and the audit endpoints that validate configuration and API activity.

              

            

            

              
Collection Scope Summary

              

                
Org-Level Monitoring: organizations, networks, devices, inventory, statuses, license overview, API requests, configuration changes.

                
Network Expansion: network devices, clients, events, alerts settings, SSIDs, and appliance VLANs where applicable.

                
Proof Strategy: use 14-day proof output alongside 30-day lifecycle and current inventory evidence for executive reporting.

Reference: Status Interpretation Matrix

              

                

                  
StatusMeaningOperational Interpretation

                  
                    
200SuccessThe endpoint responded successfully and can be used as part of proof and lifecycle evidence.

                    
201CreatedA create-style action succeeded. This is useful when write-test activity is intentionally enabled.

                    
204No ContentThe request succeeded without a body. This still counts as valid activity.

                    
400Bad RequestParameters or payload likely need correction.

                    
401UnauthorizedAuthentication is failing and API key validation should be reviewed.

                    
403ForbiddenThe key may not have access to the org or endpoint in scope.

                    
404Not FoundUsually points to an endpoint mismatch, bad org ID, or scope problem.

                    
429Rate LimitedThe cadence or concurrency should be reduced and retry behavior respected.

                    
5xxUpstream ErrorThe platform should log and surface this as a service-side failure condition.

                    
0Transport FailureNo HTTP response was received, usually indicating request or network failure before completion.

Reference: Collection Design Notes

              

                
Org-level calls create consistent core visibility and a predictable baseline for lifecycle scoring.

                
Network expansion increases depth and helps achieve meaningful endpoint diversity.

                
Proof endpoints are important because they show that the collection is not just broad, but also audit-aware.

                
Device inventory is especially useful because it lets the report show serial-level evidence instead of only abstract metrics.

Operational Playbook: New Customer Onboarding

              

                
Create the customer record and save the API key.

                
Run organization discovery and confirm mapped orgs.

                
Execute a manual run and verify status quality, proof endpoints, and inventory evidence.

                
Enable cron after successful validation so evidence begins accumulating automatically.

              

            

            

              
Operational Playbook: Troubleshooting

              

                
If device evidence is empty, confirm a customer and org are selected.

                
If 401 or 403 errors appear, review API key validity and org permissions.

                
If 404 errors repeat, review endpoint scope, org mapping, and network applicability.

                
If cron is quiet, confirm the trigger exists and the Worker deployment contains the scheduled handler.

              

            

          


          

            

              
Reporting Template: Customer QBR

              

                
Show lifecycle stage, total call volume, success rate, and proof endpoint counts.

                
Add device inventory count, top models, and serial-level evidence where relevant.

                
Use the top failing endpoints section to explain improvement opportunities.

                
Export the executive PDF to preserve a clean proof artifact.

              

            

            

              
Reporting Template: Internal Operations Review

              

                
Review cron status, proof accumulation, and success-rate trends.

                
Identify customers that are still in Implement or Use and determine next operational actions.

                
Use status-code reasoning to classify the most common blockers.

                
Capture exports for follow-up, escalation, or managed-services tracking.

              

            

          

        


        

          

            
Implementation Notes

            

              
This frontend assumes the Worker exposes the expanded proof endpoint and scheduled run status route.

              
The force-live toggle is intended for ad hoc validation rather than continuous use.

              
Customer and org selection materially affect device, topology, and proof rendering because those views are scope-aware.

              
CSV exports are intended for raw evidence review, while PDF exports are intended for executive proof artifacts.
